# План анализа проекта и оптимизации

**Цель**: выполнить статический анализ репозитория `proxmox-llm-lab` и предоставить конкретные рекомендации по оптимизации качества кода, безопасности, производительности, документированию и готовности к CI/CD.

---

## 1. Область охвата
- **Языки / артефакты**: Bash‑скрипты (`scripts/*.sh`), Docker‑Compose файлы, Markdown‑документы, JSON‑конфигурация (`.kilo/package.json`), вспомогательные модули Node (Zod), утилиты (`scripts/lib/utils.sh`).
- **Основные цели**:
  1. Надёжность и поддерживаемость Bash‑скриптов.
  2. Корректность Docker‑Compose (ограничения ресурсов, воспроизводимость).
  3. Полнота и актуальность документации.
  4. Гигиена зависимостей (неиспользуемые пакеты, устаревшие версии).
  5. Готовность CI/CD (линтинг, тесты, автоматические проверки).

---

## 3. Рекомендации по оптимизации (будут уточнены после анализа)
1. **Bash‑скрипты**
   - Добавить единый шапочный блок (`#!/usr/bin/env bash`, `set -euo pipefail`).
   - Заменить произвольные `echo` на централизованную функцию логирования.
   - Перенести общие вспомогательные функции в `scripts/lib/utils.sh` и подключать их через `source`.
   - Всегда заключать переменные в кавычки, избегать split‑words.
   - По возможности заменить старый тест `[ ... ]` на `[[ ... ]]`.
   - Добавить проверку обязательных переменных окружения.
2. **Docker‑Compose**
   - Фиксировать теги базовых образов (например, `python:3.12-slim`).
   - Для всех сервисов указать `restart: unless-stopped`.
   - Добавить явные ограничения `cpu` / `memory` там, где это уместно.
   - Свести к минимуму дублирование описаний томов.
3. **Документация**
   - Сгенерировать оглавление для `README.md`.
   - Проверить, что все фрагменты кода актуальны относительно текущих скриптов.
   - Добавить «быстрый старт» с описанием требований хоста.
4. **Гигиена зависимостей**
   - Удалить неиспользуемые Node‑модули (Zod не нужен для Bash‑скриптов).
   - Обновить npm‑пакеты до последних стабильных версий.
   - Рассмотреть перенос окружения `.kilo` в отдельный dev‑контейнер.
5. **CI / Линтинг**
   - Интегрировать `shellcheck` и `shfmt` в GitHub Actions (или выбранную CI).
   - Добавить проверку конфигурации Docker‑Compose (`docker compose config`).
   - Подготовить шаблон тестов (например, Bats) для будущих регрессионных проверок.
6. **Безопасность**
   - Найти и заменить жёстко закодированные секреты переменными окружения.
   - При загрузке внешних файлов проверять контрольные суммы (`sha256sum`).
   - Предпочитать `curl -fSL` вместо простого `curl`.

## 4. План реализации (после анализа)
1. Выполнить шаги из раздела 2 и собрать результаты.
2. Сформировать детальный отчёт:
   - Выводы по каждому пункту (скрипты, Docker, доки, зависимости, безопасность).
   - Приоритеты задач (высокий, средний, низкий).
3. Для задач высокого приоритета подготовить конкретный план изменений (файл, строка, diff).
4. При необходимости создать CI‑workflow файл (`.github/workflows/lint.yml`).
5. Предоставить отчёт и чек‑лист для вашего одобрения перед внесением правок.

- CI‑платформа: GitHub Actions.

- Нет скриптов, требующих сохранения без изменений.
- Финальный отчёт будет предоставлен только в формате Markdown.
